# System Architecture & Flow

This document details the mechanics of the orchestrator loop, the responsibilities of the generative and advisory subagents, the metrics behind the validation axes, and the automatic case-ending reconciliation loop.

---

## 1. Orchestrator Control Loop

The main pipeline loop coordinates execution using a rigid LangGraph state
machine. Its graph edges, rather than an LLM orchestration prompt, enforce
the draft → verify → route correction cycle.

1.  **Batch Ingestion**: Reads normalized verses from `dataset/inputs/` and groups them by target poetic meter to maximize prompt coherence.
2.  **Correction passes (Up to 3 passes)**:
    - The diacritizer works on non-locked, broken verses.
    - At the end of each pass, the orchestrator triggers `verify_batch_tool`.
    - Verses that pass the metrical check are immediately marked as `locked` and registered in the thread-safe local ledger (`record_locked_verse_tool`). Once locked, a verse is shielded from future passes.
3.  **Iteration Limit**: The correction loop is capped at a strict maximum of **3 passes** per batch.
    - Any verse that does not converge within 3 passes is permanently marked as `unresolved`.
    - It is routed to `log_unresolved_tool` which writes it to `dataset/verses_rejected.jsonl` with an `unresolved_max_passes` marker, as well as exporting a detailed audit log in `logs/disagreements/`.
4.  **Advisory Check**:
    - Once a batch reaches full resolution (all verses are locked or unresolved), the orchestrator triggers the batched advisory payloader (`build_batched_advisory_payload_tool`).
    - If the payload is empty, the advisory phase is skipped.
    - Otherwise, the payload is dispatched in single parallel sweeps to `irab_checker_batch` and `naturalness_critic_batch`.
    - The orchestrator runs programmatic alignment guards (`validate_advisory_batch_alignment` and `validate_naturalness_batch_alignment`) to assert 100% ID and text alignment.
    - _Fallback Mode_: If either alignment guard fails (due to LLM array mismatches or ID drift), the orchestrator logs the warning and falls back to calling standard single-verse subagents (`irab_checker` and `naturalness_critic`) sequentially.
    - The ledger is cleanly deleted via `read_ledger_tool(clear=True)` after the advisory checks complete.

## 2. Advisory & Generative Subagents

### Diacritizer (Generative)

- **Role**: Drafts and corrects diacritic marks (Harakat) for non-locked, broken verses.
- **Constraints**: Operating inside a strict boundary, it is given the target meter, locked verses (read-only), and a `correction_report` generated by `verify_batch_tool`. It has **no access** to validation or scoring functions, meaning it drafts without grading its own output. It is prohibited from altering the core consonant skeleton.

### Irab Checker (Advisory)

- **Modes**: Executed in batched mode (`irab_checker_batch`) with a fallback to sequential single-verse mode (`irab_checker`).
- **Role**: Reviews the basic grammatical legitimacy of already metrically verified verses.
- **Grammar Classification**:
  - `case_ending_swap`: Anomalies situated purely on the final short vowel of a word, where swapping the vowel solves the error without impacting the meter.
  - `structural`: Grammatical inconsistencies requiring changes to consonants, syllable lengths, or internal vowels. These represent metrically risky alterations and are treated as possible poetic licenses (الضرورات الشعرية).
- **Output Contract**: The batched version outputs an array of verdicts, including `target_word_skeleton` and `hemistich` keys, allowing the Alignment Guard to programmatically prevent word-mismatch contamination.

### Naturalness Critic (Advisory)

- **Modes**: Executed in batched mode (`naturalness_critic_batch`) with a fallback to sequential single-verse mode (`naturalness_critic`).
- **Role**: Targets the specific _Goodhart gap_ where a diacritized verse meets the mathematical patterns of `pyarud` but does so using unnatural syllable elongations, silent-letter abuse, or forced vowels that a native speaker would find implausible. Its findings are purely advisory and cannot block commits.

## 3. Four-Axis Validation Core

```
[ Proposed Verse ] ──► [ Security Gate ] ──► [ Skeleton Fidelity Gate ] ──► [ pyarud Structural Gate ] ──► [ COMMIT ]
                             │                            │                             │
                          (Reject)                     (Reject)                      (Reject)
                             ▲                            ▲                             ▲
                             └────────────────────────────┼─────────────────────────────┘
                                                          │
                                               [Programmatic Alignment Guard]
                                                          ▲
                                                          │
                                            [LLM Advisory Verdict Array]
```

To maintain absolute correctness, any proposed verse must clear three **Deciding Gates** before writing. The programmatic Alignment Guard protects the subsequent Advisory Annotation loop from LLM ID Drift:

1.  **Security Axis (Sanitizer)** `[Deciding]`: Runs `sanitize_output_tool` to scan character byte structures against allowed Arabic Unicode blocks (`0x0600`-`0x06FF`, presentational forms, common punctuation). Blocks unwanted control characters or injected payloads.
2.  **Fidelity Axis (Consonant Skeleton)** `[Deciding]`: Runs `verify_skeleton_fidelity_tool` to strip all diacritics and compare the core letters exactly against the trusted original input stored on disk under `dataset/inputs/`. Prevents semantic drift or verse substitution.
3.  **Structural Axis (pyarud)** `[Deciding]`: Re-evaluates the metrical weights using `verify_single_verse_tool` to confirm the syllable rhythm matches the target meter template.
4.  **Linguistic Axis (LLM Judgment)** `[Advisory]`: Documents grammatical or stylistic anomalies. If a verse passes the three deciding gates but carries unresolved linguistic flags, it is committed to `dataset/verses.jsonl` but annotated with `needs_review=True` and exported to `logs/disagreements/` for human auditing.

## 4. Mechanical Case-Ending Reconciliation

Because short Arabic vowels (fatha, damma, kasra) are metrically identical (representing a single moving unit / mutaharrik), a simple grammatical case-ending mismatch can often be corrected with zero metrical cost.

The system resolves this via a specialized, deterministic loop:

```
[ irab_checker_batch / irab_checker flags 'case_ending_swap' ]
│
▼
[ Programmatic Alignment Guard Validation ] ── (IF FAIL) ──► [ Fall Back to Single-Verse Sequential ]
│ (IF PASS)
▼
[ reconcile_case_ending_tool ] ──► (Apply swap on target word_index of hemistich)
│
▼
[ verify_single_verse_tool ]
│
┌────────┴────────┐
▼ (PASS)          ▼ (FAIL)
[ Commit Reconciled ]   [ Fall back to original ]
- reconciled = True     - Commit pre-swap text
- needs_review = False  - irab_flag = True
                        - needs_review = True
                        - Log disagreement
```

1.  **Detection**: The `irab_checker_batch` (or `irab_checker` in fallback) flags a verse as having a `case_ending_swap` issue, outputting the `hemistich`, `word_index`, and the recommended `target_harakah`.
2.  **Alignment Check**: The orchestrator runs `validate_advisory_batch_alignment` to ensure the targeted word consonant skeleton matches the ledger original. If validation fails, it falls back to sequential single-verse execution.
3.  **Modification**: The orchestrator invokes `reconcile_case_ending_tool` to mechanically swap the trailing short vowel of the targeted word in the specific hemistich.
4.  **Verification**: The orchestrator immediately re-runs `verify_single_verse_tool` on the modified text.
5.  **Routing**:
    - _If the recheck passes_: The reconciled text is committed with `reconciled=True`. Since the conflict was resolved programmatically, `needs_review` remains `False`.
    - _If the recheck fails_ (due to edge-case converter variations): The system falls back to the original, metrically sound pre-swap text, commits it with `irab_flag=True`, marks `needs_review=True`, and exports a disagreement record to `logs/disagreements/`.
